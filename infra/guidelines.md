# 팀모드 운영 지침

이 세션은 팀모드다. 팀의 메모리(맥락)를 적극 활용해 일한다.

## 작업 전 — 맥락을 먼저 찾는다
- 어떤 작업을 시작하기 전, 관련 맥락을 조회한다 — 팀 지식(`tm-knowledge`) + 세션로그(이전에 누가 무엇을·왜 했는지).
- 팀이 쌓아둔 제품·결정·규칙·작업 이력을 모르고 진행하지 않는다.

## 메모리가 비어 있으면 — 채우게 요구한다
- 팀 메모리에 맥락이 부족하면, 사용자에게 채워달라고 요구한다:
  > "이 작업을 제대로 하려면 팀 맥락이 필요해요 — X를 메모리에 넣어주실 수 있나요?"
- 빈약한 맥락으로 추측해 진행하지 않는다.
- **기존 문서가 있으면 memory로 옮길 것을 적극적으로 권한다** — 노션·구글독스·컨플루언스 등에 이미 써둔 게 있으면 그걸 가져오는 게 제일 빠르다. "혹시 이미 정리된 문서 있으세요? 있으면 바로 옮겨드릴게요"라고 먼저 물어본다.
- **`product/tech/`가 비어있으면**: GitHub 레포 링크를 요청하고, 받으면 백그라운드 서브에이전트로 레포를 분석해 `stack.md`·`features.md`를 자동으로 채운다.

## 핵심 맥락 (팀이 갖춰야 할 것)
- **제품·브랜드**: 무엇을 만드나, 핵심 기능, 핵심 고객, 브랜드 철학(차별화·가치관), 로드맵
- **그라운드룰**: 팀이 일하는 방식, 코드 컨벤션, 지켜야 할 원칙
- **결정사항**: 왜 그렇게 정했나 (반복 논의 방지)
- **아키텍처·기술 스택**: 시스템 구조, 핵심 기술(DB·프레임·언어), 제약 사항

## 메모리는 동사로 쓴다
- `memory/`를 직접 Edit/Write 하지 않는다 (INDEX·커밋 절차가 빠진다).
- 조회 `tm-knowledge` / 추가·수정 `tm-manage-knowledge`.
- 사용자가 팀에 남길 결정·지식을 말하면 `tm-manage-knowledge`로 쌓는다. 쌓인 지식은 다음 세션에 자동 주입된다.

## teammode 자체에 문제가 보이면 — 본레포에 제보를 제안한다
- teammode 제품(`infra/`)의 버그·불편·개선점을 발견하면 묻어두지 말고, 사용자에게 제보를 제안한다:
  > "이거 teammode 본레포에 제보할까요?"
- 제보는 `tm-contribute` 스킬로 — 먼저 **본레포 대조 진단**으로 "진짜 본레포 버그인지, 우리 인스턴스의 로컬 문제인지"를 가린다(헛이슈 방지). 인스턴스 문제면 `tm on`(sync)을 권하고, 본레포 버그면 GitHub 이슈로 올린다.
- 우리가 겪은 문제는 다른 팀에게도 도움이 된다 — 제품이 같이 자란다.
