## 1. 목적과 범위

이 문서는 teammode 팀 메모리의 **데이터 표준**을 정의한다: 디렉토리 구조, 세션로그 파일 포맷, 컨텍스트 주입 규칙, 버저닝. 어떤 에이전트(Claude Code/Codex/Hermes/…)로 읽고 쓰든, 이 표준을 따르는 한 같은 팀 메모리를 공유할 수 있다 — 크로스에이전트 호환의 토대가 되는 스펙이다.

훅 등록·어댑터 등 **실행 계층**은 [스펙 02](02-hook-manifest.md)가, 독립 구현의 호환 선언은 [스펙 03](03-conformance.md)이 다룬다.

### 1.1 표기 규약

- **필수** — 따르지 않으면 이 스펙 비준수. conformance 검사 대상.
- **권장** — 따르지 않아도 비준수는 아니나, 정당한 사유가 있어야 한다.
- **예약** — v0.1에서는 자리만 정의하고 의미를 확정하지 않은 항목. 임의 사용 금지.

### 1.2 용어

| 용어 | 정의 |
|---|---|
| **팀 레포** | 팀 메모리와 엔진 설정을 담는 git 저장소. 팀당 1개, private. |
| **팀 루트** | 팀 레포의 로컬 클론 경로. 구현은 **런타임 훅**이 참조할 환경변수를 제공해야 한다(필수) — 런타임 훅은 에이전트 하니스가 발동하므로 인자를 받을 통로가 없기 때문이다. 변수명은 구현 세부사항이다(현행 reference 구현: `TEAMMODE_HOME`). **단, 의도적으로 호출되는 엔진/어댑터(on/off·설치)는 팀 루트를 명시 인자(예: `--root`)로만 받아야 하며 환경변수를 신뢰해서는 안 된다(필수)** — ambient env 신뢰는 SubprocessEngine 격리를 우회한 직접 호출 시 엉뚱한(호스트) 폴더를 건드리는 사고의 근본 원인이다(§2.4 참조). |
| **세션로그** | 멤버가 에이전트 세션에서 수행한 팀 작업의 일지. 본 스펙 §3. |
| **작업일** | 06시 컷 적용 후의 날짜. 본 스펙 §3.2. |
| **주입(injection)** | 세션 시작 시 팀 메모리 일부를 에이전트 컨텍스트에 자동 로드하는 것. |

## 2. 디렉토리 구조

팀 메모리는 팀 루트의 `memory/` 아래에 있다. 위치는 **필수**, 구조는 코어(필수)·예약·팀 확장(자유)으로 나뉜다.

### 2.1 코어 구조 (필수)

```
memory/
├── INDEX.md                      # 메모리 인덱스 — 세션 시작 시 주입되는 단일 진입점
└── team/
    ├── members.md                # 멤버 명부 — 영문 이름(소문자)·역할·연락 정보의 단일 소스
    ├── sessions/<이름>/          # 멤버별 세션로그 (§3). <이름>은 members.md의 영문 이름
    ├── decisions/                # 확정된 결정사항
    │   ├── current.md            #   활성 결정
    │   └── archive/              #   과거 결정
    └── meeting/
        ├── summary/              # 회의록 요약본
        └── raw/                  # 회의 원본 (STT·텍스트)
```

- **INDEX.md** — 폴더별 "여기에 넣는 것" 설명을 표로 유지한다. 새 폴더를 만들면 INDEX.md 갱신이 **필수**다. 등재되지 않은 폴더는 주입·탐색 대상에서 누락된다.
- **members.md** — 멤버의 영문 이름은 시스템 계정명(`$USER`)이 아니라 이 파일에 등재된 이름이다. 코드·훅·스킬은 멤버 이름을 하드코딩하지 말고 이 파일을 참조해야 한다(필수). 영문 이름은 소문자로 쓰며 팀 내에서 고유해야 한다(필수 — 폴더명·frontmatter가 이 이름을 그대로 쓴다). 파일의 상세 포맷(필드 구성·마크다운 구조)은 v0.2에서 확정한다(예약).
- **sessions/<이름>/** — 세션로그(`YYYY-MM-DD.md`, §3) 외의 보조 파일(예: 도구 설정 캐시)을 둘 수 있다. 보조 파일은 주입 대상이 아니다. 단, **`YYYY-MM-DD`로 시작하는 `.md` 파일명은 세션로그 네임스페이스로 예약**되어 보조 파일에 쓸 수 없다(`-late` 등 분할 파일은 §3.1 위반으로 검사 대상). 그 외 파일명의 존재를 conformance 검사가 위반으로 판정해서는 안 된다.
- **decisions/** — "확정된" 결정만 들어간다. 논의 중인 사안은 세션로그·회의록에 머문다.

### 2.2 권장·예약 항목

| 경로 | 분류 | 내용 |
|---|---|---|
| `team/reviews/` | 권장 | 외부 평가·피드백. 파일명 `YYYY-MM-DD-출처-단계.md` |
| `team/ground-rules.md` | 권장 | 팀 운영 그라운드 룰. 엔진(다이제스트 등)이 참조할 수 있는 표준 위치 |
| `banner.txt` | 권장 | 팀 배너 캐시의 표준 위치. `team.config.json`의 `banner_file`이 가리키며, 생성·갱신 방법은 구현 세부 |

### 2.3 팀 확장 (자유)

팀은 `memory/` 아래에 자유롭게 폴더를 추가할 수 있다(예: reference 구현의 `product/`, `extras/`). 규칙 두 가지:

1. **기존 폴더로 충분하면 새 폴더를 만들지 않는다** (폴더 증식 방지 — 권장).
2. 새 폴더는 INDEX.md에 등재한다 (필수).

### 2.4 쓰기 위치 규칙 (필수)

팀 메모리 쓰기는 항상 **팀 루트의 `memory/`** 에 한다. 현재 작업 중인 코드 레포에 우연히 존재하는 `./memory/`에 쓰는 것은 금지다. 구현은 세션 리마인더 등으로 이 혼동을 방지하는 것을 권장한다.

**엔진의 팀 루트 해석(필수)**: on/off 등 상태를 바꾸는 엔진 호출은 대상 팀 루트를 **명시 인자로만** 받는다. 환경변수 폴백·cwd 추측을 해서는 안 되며, 인자가 없으면 **동작하지 말고 에러로 종료**한다. 근거: ambient 환경변수(예: 호스트 toolkit을 가리키는 변수)가 새어들면, 격리 하니스를 우회한 직접 CLI 호출이 호스트의 상태 마커(`.acme-active` 등)를 삭제하는 실사고가 발생했다. 엔진이 "어느 폴더를 건드릴지 추측하지 않게" 하는 것이 근본 처방이다. 마찬가지로 에이전트 설정 파일 경로(예: `~/.claude/settings.json`)도 명시(격리 경로 또는 명시적 설치 플래그) 없이 실 사용자 설정에 쓰지 않는다.

## 3. 세션로그 표준

### 3.1 파일 위치와 단위 (필수)

- 경로: `memory/team/sessions/<이름>/YYYY-MM-DD.md`
- **하루 1파일.** 같은 작업일의 로그는 기존 파일에 이어 쓴다. `-late`, `-2` 등 분할 파일 생성 금지.
- `YYYY-MM-DD`는 작업일(§3.2)이며 frontmatter의 `date`와 일치해야 한다.

### 3.2 06시 컷 (필수)

작업일의 경계는 자정이 아니라 **06:00(팀 timezone, `team.config.json`의 `team.timezone`)** 이다.

- 00:00~05:59에 작성을 시작하는 로그 → **전날** 날짜 파일에 이어 쓴다.
- 06:00 이후 → 당일 파일.
- **판정 시점은 세션에서 그 로그 작성을 시작한 시각**이다. 세션 도중 06:00을 넘겨도 시작 시점의 작업일 파일에 계속 이어 쓴다 — 한 흐름의 작업을 두 파일로 찢지 않는다.

근거: 새벽 작업은 의미상 전날 세션의 연장이다. 자정 컷은 한 흐름의 작업을 두 파일로 찢는다.

### 3.3 frontmatter (필수)

```markdown
---
author: jane-doe
date: 2026-06-11
summary: 훅 어댑터 레이어 설계 확정 — events.json 번역표, 폴백 정책, 서비스 추상화까지
---
```

| 필드 | 필수 | 정의 |
|---|---|---|
| `author` | 필수 | members.md에 등재된 영문 이름. `$USER` 등 시스템 계정명 금지 |
| `date` | 필수 | 작업일 (06시 컷 적용). 파일명과 일치 |
| `summary` | 필수 | 그날 작업의 **한 줄** 요약. **v0.1 신규 필드** (아래 참조) |

위 3필드는 필수 최소 집합이며, 팀·구현이 추가 필드를 두는 것은 허용된다.

**`summary` 필드** — 스케일 주입(§4)·get-context 요약·대시보드·충돌 레이더(이상 둘은 로드맵)가 공용으로 읽는 필드다. 하루 중 작업 내용이 바뀌면 그날의 대표 내용으로 **갱신**한다(append가 아니라 교체). 한 줄(권장 100자 이내), 그날 로그를 안 읽은 동료가 "무슨 작업이었는지"를 파악할 수 있게 쓴다.

> 마이그레이션: v0.1 이전에 작성된 `summary` 없는 로그는 비준수로 취급하지 않는다. 단 신규 작성분부터는 필수다.

### 3.4 상세도 기준 (필수)

세션로그는 "한 일 목록"이 아니다. **나중에 읽어도 그때의 맥락을 완전히 복원할 수 있을 것** — 이것이 단일 기준이다. 각 작업 항목은 다음을 한 흐름으로 담는다:

1. **무엇을 했나** — 결과물·변경 사항
2. **왜 그렇게 결정했나** — 근거
3. **접은 대안** — 검토했으나 버린 선택지와 버린 이유
4. **막힌 점** — 미해결 블로커, 우려
5. **다음 단계** — 이어서 할 일

추가 규칙:

- **팀 작업만.** 개인 일정·사적 내용은 제외한다(필수).
- 같은 날 여러 세션이 있으면 시각 구분선(예: `## 14:30 —`)으로 이어 쓴다(권장).
- 구현은 일정 시간(reference: 30분) 이상 로그 미갱신 시 세션 내 리마인드를 제공하는 것을 권장한다. 규율을 사람의 습관이 아니라 훅이 들게 하는 것이 teammode의 핵심 설계이기 때문이다.

## 4. 주입 규칙 (스케일)

세션 시작 시 구현은 팀 메모리를 컨텍스트에 주입한다. 주입량은 팀 크기에 따라 달라진다 — 전원 전문(全文) 주입은 인원에 비례해 컨텍스트가 폭발하기 때문이다.

### 4.1 규모별 규칙 (필수)

| 팀 크기 | 세션로그 주입 방식 |
|---|---|
| **~4인** | 전원 최근 로그 **전문** 주입 |
| **5인 이상** | 전원 `summary` 한 줄 + **본인** 로그 전문 + 동료 상세는 **lazy load** (필요 시 파일을 직접 읽음) |

- **팀 크기** = members.md에 등재된 멤버 수.
- **주입의 기본 단위** = 멤버별 **가장 최근 작업일의 파일 1개** (전문이든 summary든 동일). 구현은 주입 범위를 더 넓게(최근 N일치 등) 제공할 수 있으나, 기본값은 1파일이어야 한다.
- 대상 파일에 `summary`가 없으면(§3.3 마이그레이션 단서의 구 로그) 그 멤버의 **summary 주입은 생략**한다 — 전문 폴백 주입 금지 (컨텍스트 폭발 방지가 이 절의 목적이므로).

`summary` 필드(§3.3)가 필수인 이유가 이것이다 — 5인+ 팀에서 summary가 비면 그 멤버의 작업이 동료 시야에서 사라진다.

### 4.2 `groups` (예약)

`team.config.json`의 `groups` 키는 **v0.1에서 예약어**다. 스쿼드 단위 주입 범위(같은 그룹은 전문, 다른 그룹은 summary)를 위한 자리이며, 의미 확정은 v0.2에서 한다. v0.1 구현은 이 키를 무시해야 하며, `null` 외의 값에 의미를 부여해서는 안 된다.

### 4.3 적용 범위 주의

v0.1의 타겟 팀 크기는 **2~5인**이다. 7인 이상에서의 동작은 본 스펙이 보증하지 않는다.

## 5. 버저닝

- 스펙 01~03은 **한 묶음으로 공통 `spec_version`을 공유**하며, 현재 **0.1**이다. 묶음 내 어느 문서든 규범 변경이 생기면 묶음 전체의 버전이 함께 오른다. 팀 레포의 `team.config.json` 최상위 `spec_version` 필드는 그 팀 데이터가 따르는 **스펙 묶음** 버전을 선언한다.
- **변경 절차**: 포맷 필드의 추가·의미 변경·폴더 구조 변경은 minor bump(0.1 → 0.2)와 CHANGELOG 기록을 필수로 한다. CHANGELOG는 스펙을 배포하는 본진 레포에서 관리한다(팀 레포가 아니다). 오탈자·설명 보강 등 의미 불변 수정은 버전을 올리지 않는다.
- **0.x 기간에는 하위 호환이 깨질 수 있다.** 독립 구현이 2개 이상 생기면 1.0으로 동결하고 RFC-lite 변경 절차(제안 이슈 → 구현 영향 검토 → 합의 후 머지)를 도입한다.
- 구현은 자신이 지원하는 `spec_version`을 명시해야 한다([스펙 03](03-conformance.md) §4의 등재 요건).

## 부록 A. 초안 대비 확정 사항

정식판에서 초안·운영 관행 대비 새로 확정하거나 일반화한 것:

| 항목 | 결정 |
|---|---|
| `summary` 갱신 방식 | append가 아니라 그날 대표 내용으로 교체 |
| `summary` 길이 | 한 줄, 권장 100자 이내 |
| 기존 로그의 `summary` 부재 | 비준수로 취급하지 않음 (신규 작성분부터 필수) |
| frontmatter 추가 필드 | 필수 3필드 외 추가 필드 허용 |
| 06시 컷 판정 시점 | 로그 작성 시작 시각 기준 — 세션 도중 06시를 넘겨도 파일을 찢지 않음 |
| 영문 이름 표기 | 소문자·팀 내 고유 (폴더명·frontmatter 일관성). members.md 상세 포맷은 v0.2 예약 |
| `sessions/<이름>/` 보조 파일 | 허용 (주입·conformance 검사 대상 아님). 단 `YYYY-MM-DD*` `.md`는 세션로그 네임스페이스로 예약 |
