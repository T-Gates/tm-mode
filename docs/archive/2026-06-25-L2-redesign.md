# L2 Redesign — MCP Registrar (Confirmed 2026-06-25)

> Design/implementation reference document (archive, not authoritative). The authoritative specs live in `docs/spec/`. This document is the source of truth for those edits.
> Decision source: the 2026-06-21 "standard toolset" decision → simplified and confirmed during the 6/24-25 brainstorming session (formerly the `docs/BACKLOG.md` L2 section — this document and `docs/spec/` are now the preserved record).

## One Line

**L2 = a registrar that plugs official vendor MCPs into slots.** tm-mode only *connects* them; actions such as issue creation or calendar event creation are performed by the AI directly calling those MCP tools.

## Key Decisions (Option A Confirmed)

| Item | Decision |
|------|------|
| tm-mode role | **Connection only**. Plug the MCP into the slot and stop there |
| Actions | The AI directly calls `mcp__<alias>__<벤더도구>` (tm-mode does not wrap it) |
| Discarded | `role_server` proxy, hand-written `handlers/<role>.py`, role abstraction, and the "reuse > absorb > custom" priority decision |
| Discarded | The personal MCP concept (too much for teams of 3-7; keep everyone unified) |

> **Option A vs Option B** (confirmed: A):
> - A = tm-mode only connects MCPs. For calendar additions, the AI calls the Notion MCP tool directly.
> - B = wrap the MCP one more layer with action commands such as `tm-calendar add` → **discarded because it revives the role_server abstraction**.

## Slots & Alias

```
슬롯(역할)   등록 alias        실제 내용물
issues    →  (벤더 MCP)     =  팀이 고른 공식 MCP (linear / 노션칸반 ...)
chat      →                =  slack ...
docs      →                =  ...
calendar  →                =  google / 노션캘린더 ...
```

- Slots = issues / chat / docs / calendar.
- One provider may serve multiple slots (one Notion MCP → issues=Kanban, calendar=calendar). Register the MCP once and add N config mappings.
- Whether to standardize the tool-call prefix (alias = tm-<slot>) is decided during implementation, but **do not create action wrappers (A)**.
  - Implementation confirmed: the alias is **`tm-<provider>`** (provider-based, not slot-based, e.g. `linear`→`tm-linear`). This creates a teammode namespace that can coexist with a user MCP of the same name. At runtime, normalize maps the tool name `mcp__tm-<provider>__<도구>` back to the canonical server name (authoritative spec: `internals.md` §2.8/§6.1).

## tm-connect Flow

```
slot 선택 (issues/chat/docs/calendar)
  ↓
config.services[slot].provider 있나?
  없음(첫등록자):
    "provider 뭐?" → 검색해 후보 N개 제시
    → 유저 택 (공식 식별 = 보안 게이트)
    → 공식 MCP 레포 가져와 본 레포에 둠 (없으면 AI 자작)
    → config 기록 + GitHub push (팀 공유 선언)
  있음(후속멤버):
    config 읽음. 재선택 X.
  ↓
멤버 에이전트(claude/codex)에 그 MCP 등록 (install-mcp)
  ↓
토큰 = 각자 입력 → 로컬 금고 (0600)
  ↓
노션칸반(issues)·노션캘린더(calendar)처럼 DB 지정 필요한 슬롯만 dbid 물음
```

## MCP Provisioning

```
provider 정함
  → 공식 MCP 레포 있음 → git 가져와 본 레포 infra/mcp/<provider>/ 에 둠 + 커밋 (팀 공유 보관소)
  → 공식 없음          → AI 자작 (유저한테 안 미룸. "자체제작 X" 원칙은 폐기)
```

**Official first.** A custom implementation is only the fallback when no official MCP repo exists.

### When There Is No Official MCP — 7-Step AI Custom Build

```
1. provider API 스펙 확보 — 그 벤더의 공식 REST/GraphQL 문서를 출처로 삼는다.
2. Python MCP SDK로 MCP 서버 작성 — 그 슬롯 역할에 필요한 도구만 노출한다.
   (예: calendar 슬롯이면 list_events / create_event 정도. 슬롯 전부를 덮는 만능 X)
3. infra/mcp/<provider>/ 에 둠 — 서버 코드 + 실행 메타(기동 command).
   공식 MCP를 가져왔을 때와 같은 위치·구조다.
4. 토큰 — 공식 MCP와 동일 경로(env / 로컬 금고 0600). 자작이라고 별도 경로 X.
5. 본 레포에 커밋 = 팀 공유 보관소. 다음 멤버는 그대로 재사용한다(재자작 X).
6. 등록 — install-mcp가 공식/자작 구분 없이 동일하게 정규 서버명 alias로 등록한다(§internals 2.8).
7. 품질 — 첫 자작 직후 적대검수(서브에이전트)로 노출 도구가 실제 동작하는지 검증한다.
```

**Core principle — a custom MCP is not a role abstraction.**

- A custom MCP is **a vendor-specific MCP** that fills the official MCP slot. It exposes tools wrapping that provider API *as-is*.
- It **does not** create a unified slot interface with role verbs such as `issues_create`. That would revive the discarded `role_server`/role abstraction from Option B.
- tm-mode still only **connects (registers)** custom MCPs as well (keep Option A). Actions are performed by the AI directly calling the registered `mcp__<alias>__<벤더도구>`.

## config

```json
{
  "services": {
    "issues":   { "provider": "linear" },
    "calendar": { "provider": "notion", "database_id": "..." }
  }
}
```

- Provider mapping is the default. Tokens do not go in config (vault).
- Instance values such as dbid are used **only for slots that need them** (Notion-like providers). Slack uses all channels, so omit them.
- Whether instance values are required is declared by the provider pack (resource_fields).

## Unification / Scope

- If the team chooses Notion Calendar, **everyone uses Notion Calendar**. No separate personal variants.
- Scope: **open source for five-person teams = general distribution + simple defaults.** When unsure, choose simplicity.
