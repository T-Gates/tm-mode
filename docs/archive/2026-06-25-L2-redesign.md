## 한 줄

**L2 = 슬롯에 공식 벤더 MCP를 꽂아주는 등록기.** tm-mode는 *연결*만 하고, 동작(이슈 생성·일정 추가 등)은 AI가 그 MCP 도구를 직접 호출한다.

## 핵심 결정 (A안 확정)

| 항목 | 결정 |
|------|------|
| tm-mode 역할 | **연결만**. MCP를 슬롯에 꽂고 끝 |
| 동작 | AI가 `mcp__<alias>__<벤더도구>` 직접 호출 (tm-mode가 래핑 X) |
| 폐기 | `role_server` 프록시, 수제 `handlers/<role>.py`, 역할 추상화, "재사용>흡수>수제" 우선순위 판정 |
| 폐기 | 개인 MCP 개념 (3~7인엔 과함 — 전원 통일) |

> **A안 vs B안** (확정: A):
> - A = tm-mode는 MCP 연결만. 일정추가는 AI가 노션 MCP 도구 직접 호출.
> - B = `tm-calendar add` 같은 동작 명령으로 MCP를 한 겹 더 감쌈 → **role_server 추상화 부활이라 폐기**.

## 슬롯 & alias

```
슬롯(역할)   등록 alias        실제 내용물
issues    →  (벤더 MCP)     =  팀이 고른 공식 MCP (linear / 노션칸반 ...)
chat      →                =  slack ...
docs      →                =  ...
calendar  →                =  google / 노션캘린더 ...
```

- 슬롯 = issues / chat / docs / calendar.
- 한 provider가 여러 슬롯 가능 (노션 1개 → issues=칸반, calendar=캘린더). MCP 등록 1번, config 매핑 N줄.
- 도구 호출 prefix 통일 여부(alias = tm-<slot>로 표준화)는 구현 시 결정 — 단 **동작 래퍼는 안 만든다(A)**.

## tm-connect 흐름

```
slot 선택 (issues/chat/docs/calendar)
  ↓
config.services[slot].provider 있나?
  없음(첫등록자):
    "provider 뭐?" → 검색해 후보 N개 제시
    → 유저 택 (공식 식별 = 보안 게이트)
    → 공식 MCP 레포 가져와 본 레포에 둠 (없으면 AI 자작)
    → config 기록 + GitHub push (팀 공유 선언)
  있음(후속멤버):
    config 읽음. 재선택 X.
  ↓
멤버 에이전트(claude/codex)에 그 MCP 등록 (install-mcp)
  ↓
토큰 = 각자 입력 → 로컬 금고 (0600)
  ↓
노션칸반(issues)·노션캘린더(calendar)처럼 DB 지정 필요한 슬롯만 dbid 물음
```

## MCP 마련

```
provider 정함
  → 공식 MCP 레포 있음 → git 가져와 본 레포 infra/mcp/<provider>/ 에 둠 + 커밋 (팀 공유 보관소)
  → 공식 없음          → AI 자작 (유저한테 안 미룸. "자체제작 X" 원칙은 폐기)
```

**공식 우선.** 자작은 공식 MCP 레포가 없을 때만의 대안이다.

### 공식 없을 때 — AI 자작 7단계

```
1. provider API 스펙 확보 — 그 벤더의 공식 REST/GraphQL 문서를 출처로 삼는다.
2. Python MCP SDK로 MCP 서버 작성 — 그 슬롯 역할에 필요한 도구만 노출한다.
   (예: calendar 슬롯이면 list_events / create_event 정도. 슬롯 전부를 덮는 만능 X)
3. infra/mcp/<provider>/ 에 둠 — 서버 코드 + 실행 메타(기동 command).
   공식 MCP를 가져왔을 때와 같은 위치·구조다.
4. 토큰 — 공식 MCP와 동일 경로(env / 로컬 금고 0600). 자작이라고 별도 경로 X.
5. 본 레포에 커밋 = 팀 공유 보관소. 다음 멤버는 그대로 재사용한다(재자작 X).
6. 등록 — install-mcp가 공식/자작 구분 없이 동일하게 정규 서버명 alias로 등록한다(§internals 2.8).
7. 품질 — 첫 자작 직후 적대검수(서브에이전트)로 노출 도구가 실제 동작하는지 검증한다.
```

**핵심 원칙 — 자작 MCP는 역할 추상화가 아니다.**

- 자작 MCP는 공식 MCP 자리를 메우는 **그 벤더 전용 MCP**다. 그 provider API를 감싼 도구를 *그대로* 노출한다.
- 슬롯 통일 인터페이스(`issues_create` 같은 역할 동사)를 만드는 게 **아니다**. 그건 폐기한 `role_server`/역할 추상화(B안)의 부활이다.
- tm-mode는 자작 MCP에 대해서도 여전히 **연결(등록)만** 한다(A안 유지). 동작은 AI가 등록된 `mcp__<alias>__<벤더도구>`를 직접 호출한다.

## config

```json
{
  "services": {
    "issues":   { "provider": "linear" },
    "calendar": { "provider": "notion", "database_id": "..." }
  }
}
```

- provider 매핑이 기본. 토큰은 config에 안 씀 (금고).
- 인스턴스값(dbid 등)은 **필요한 슬롯만** (노션류). slack은 채널 다 써서 생략.
- 인스턴스값 필요 여부 = provider 팩이 선언 (resource_fields).
